# ADR-007 — Row Level Security con SET LOCAL en Neon Serverless

- **Status**: IMPLEMENTED ✅ (Fase 13D, 2026-04-06 — commit `0ada843`)
- **Author**: arch-agent
- **Criticality**: SECURITY — defense-in-depth layer over ADR-004 application-level companyId checks

## Context

ContaFlow runs on Neon serverless with PgBouncer in **transaction mode**. This means:
- Session-level `SET` commands (`SET app.current_company_id = '...'`) do NOT survive the pool — each query may land on a different connection.
- Session-level advisory locks (LL-005) are also unusable for the same reason.

RLS was planned post-Fase 17 to add a database-level enforcement layer so that even if application code omits `companyId` from a query, the DB policy rejects the cross-tenant access.

## Decision

**Use `set_config(key, value, is_local := true)` (= `SET LOCAL`) inside `prisma.$transaction`.**

`SET LOCAL` applies only for the duration of the current transaction, not the session. Since Neon's PgBouncer returns the connection to the pool at transaction end, `SET LOCAL` is automatically cleaned up at `COMMIT` or `ROLLBACK`. This makes it fully compatible with transaction-mode pooling.

### Canonical pattern

```typescript
// src/lib/prisma-rls.ts
export async function withCompanyContext<T>(
  companyId: string,
  tx: Prisma.TransactionClient,
  fn: (tx: Prisma.TransactionClient) => Promise<T>
): Promise<T> {
  if (!companyId) {
    throw new Error("withCompanyContext: companyId no puede estar vacío");
  }
  // is_local=true → equivalent to SET LOCAL — cleared at end of transaction
  await tx.$executeRaw`SELECT set_config('app.current_company_id', ${companyId}, true)`;
  return fn(tx);
}
```

Every domain `$transaction` block must wrap its body with `withCompanyContext`:

```typescript
await prisma.$transaction(async (tx) =>
  withCompanyContext(companyId, tx, async (tx) => {
    // queries here execute under RLS policy
  })
);
```

### Database policies

RLS is enabled on all 14 domain tables. The policy reads the transaction-local parameter:

```sql
-- Example: applied to every domain table (fail-closed — la implementación real)
ALTER TABLE "BankTransaction" ENABLE ROW LEVEL SECURITY;
CREATE POLICY company_isolation ON "BankTransaction"
  USING      ("companyId" = current_setting('app.current_company_id', true));
-- + WITH CHECK idéntico desde 20260611_rls_force_with_check
```

> **Corrección (hallazgo externo ADR — fail-closed).** La versión borrador de este
> ADR proponía un escape `OR current_setting(...) IS NULL OR ... = ''` para que
> migraciones/admin funcionaran sin contexto. **Ese escape nunca se implementó** y
> sería *fail-open*: una query bajo `authenticated` sin `withCompanyContext` vería
> filas de TODOS los tenants — justo el escenario que la RLS debe blindar. La
> migración real (`20260406110000_fase13d_rls_company_isolation`) y `ADR-007-rls-row-level-security.md`
> ya usan la policy sin escape. Con `missing_ok = true`, si falta el contexto
> `current_setting` devuelve NULL y `"companyId" = NULL` evalúa a NULL → la fila
> **no** es visible (*fail-closed*, 0 filas). Migraciones/admin no necesitan el
> escape porque corren bajo `neondb_owner` (BYPASSRLS).

## Rejected alternatives

| Alternative | Reason for rejection |
|---|---|
| `SET app.current_company_id = '...'` (session-level) | Does not survive PgBouncer transaction mode — value leaks across requests (LL-005 analogue) |
| Neon direct connection for all queries | Defeats the purpose of the pooler; latency +40ms per query |
| Application-only enforcement (ADR-004) | Single layer — if one query misses `companyId`, data leaks. RLS is a mandatory second layer |
| Supabase RLS with `auth.uid()` | Supabase-specific, not available on Neon |

## Consequences

- **Positive**: Defense-in-depth — a bug in application code that omits `companyId` is still blocked at DB level
- **Positive**: `SET LOCAL` is transaction-scoped → zero risk of cross-request contamination in the pool
- **Negative**: Every domain `$transaction` must call `withCompanyContext` — if forgotten, the RLS policy uses `NULL` setting and the query returns **0 rows** (*fail-closed*, no leak). The `company-isolation.test.ts` architectural test mitigates the missing-data case at the action layer.
- **Operational rule**: `withCompanyContext` is mandatory in every `prisma.$transaction` that touches a domain table. Migrations and admin scripts run outside `$transaction` and are exempt.
